Styling in React: Multiple Approaches

Unlike traditional HTML/CSS, React offers multiple ways to style components. Each approach has its strengths, and understanding when to use each is key to writing maintainable code.

üéØ Why Multiple Styling Methods?

React's component-based architecture means styles can be scoped, dynamic, and co-located with components. This solves many traditional CSS problems like global namespace pollution and specificity wars.

üîë Key Differences from Traditional CSS

‚ùå Traditional HTML/CSS

<!-- HTML -->
<div class="card">...</div>
<div class="card active">...</div>

/* CSS - GLOBAL! */
.card { ... }
.card.active { ... }
  • Uses class attribute
  • Styles are global
  • String-based class names

‚úÖ React Styling

{/* JSX */}
<div className="card">...</div>
<div className={`card ${isActive ? 'active' : ''}`}>

{/* Or inline */}
<div style={{ color: isActive ? 'green' : 'gray' }}>
  • Uses className (not class!)
  • Can be scoped per component
  • Dynamic with JavaScript
‚ö†Ô∏è Remember: className, not class!
In JSX, use className instead of class because class is a reserved word in JavaScript.

Part 2: Inline Styles - JavaScript Style Objects
‚ñº

Inline Styles in React

Inline styles in React use JavaScript objects instead of CSS strings. This allows for dynamic styling based on props and state.

üìê Syntax Rules

Key Differences from CSS:
  • camelCase property names: background-color ‚Üí backgroundColor
  • String values for most properties: color: 'red'
  • Numbers default to pixels: padding: 16 ‚Üí 16px
  • Double curly braces: style={{ }} (object inside JSX expression)

üìù Basic Inline Styles

  Inline Style Examples
// Method 1: Direct inline (for simple cases)
function Card() {
  return (
    <div style={{
      backgroundColor: '#1a1a2e',
      padding: 20,                  // 20px
      borderRadius: 12,              // 12px
      border: '1px solid #333',
      boxShadow: '0 4px 6px rgba(0,0,0,0.3)'
    }}>
      <h2 style={{ color: '#22d3ee', marginBottom: 8 }}>
        Card Title
      </h2>
      <p style={{ color: '#94a3b8' }}>Card content</p>
    </div>
  );
}

// Method 2: Style object variable (cleaner for complex styles)
function Card() {
  const cardStyle = {
    backgroundColor: '#1a1a2e',
    padding: 20,
    borderRadius: 12,
    border: '1px solid #333'
  };

  const titleStyle = {
    color: '#22d3ee',
    fontSize: 24,
    fontWeight: 600,
    marginBottom: 8
  };

  return (
    <div style={cardStyle}>
      <h2 style={titleStyle}>Card Title</h2>
    </div>
  );
}

üîó Using clsx or classnames Library

For complex conditional class names, use the clsx or classnames library:

  Using clsx for conditional classes
import clsx from 'clsx';
import './StudentCard.css';

function StudentCard({ student, isSelected, isHighlighted, isDisabled }) {
  // Much cleaner than string concatenation!
  const cardClasses = clsx(
    'student-card',           // Always applied
    {
      'selected': isSelected,   // Applied if isSelected is true
      'highlighted': isHighlighted,
      'disabled': isDisabled
    }
  );

  // Result: "student-card selected highlighted" (based on conditions)

  return (
    <div className={cardClasses}>
      ...
    </div>
  );
}
‚ö†Ô∏è The Big Problem: Global Scope!
CSS files are GLOBAL. If you have .card in two different CSS files, they WILL conflict!

File1.css: .card { background: red; }
File2.css: .card { background: blue; }
Result: One will override the other - unpredictable! üò±

‚úÖ Pros & Cons

  • Easy to copy existing CSS
‚ùå Disadvantages
  • Global scope (name collisions!)
  • Need unique class names
  • Dead CSS hard to track
  • No dynamic values from JS
  • Separate file to maintain

Part 4: CSS Modules - Scoped Styles (Recommended!)
‚ñº

CSS Modules: The Best of Both Worlds

CSS Modules give you all CSS features with automatic local scoping. Class names are transformed to unique identifiers, preventing conflicts.

üéØ How CSS Modules Work

You write: .card { }
Browser sees: .StudentCard_card_x7yz3 { }
Each class gets a unique hash - no more conflicts!

// Import as an object!
import styles from './StudentCard.module.css';
import clsx from 'clsx';

function StudentCard({ student, isSelected }) {
  // Access classes as object properties
  const cardClasses = clsx(
    styles.card,
    { [styles.selected]: isSelected }
  );

  // Dynamic grade class
  const gradeClass = styles[`grade${student.grade}`];

  return (
    <div className={cardClasses}>
      <div className={styles.header}>
        <h3 className={styles.name}>
          {student.name}
        </h3>
        <span className={clsx(styles.badge, gradeClass)}>
          {student.grade}
        </span>
      </div>
      <p className={styles.details}>
        Age: {student.age}
      </p>
    </div>
  );
}

export default StudentCard;

// Must manually toggle classes!
let selectedCards = new Set();

function toggleSelect(id) {
  const card = document
    .getElementById(`card-${id}`);
  
  if (selectedCards.has(id)) {
    selectedCards.delete(id);
    card.classList.remove('selected');
  } else {
    selectedCards.add(id);
    card.classList.add('selected');
  }
  
  updateSelectButton(card, id);
}

function updateSelectButton(card, id) {
  const btn = card.querySelector('.btn-edit');
  btn.textContent = selectedCards.has(id) 
    ? 'Deselect' 
    : 'Select';
  btn.classList.toggle(
    'btn-selected', 
    selectedCards.has(id)
  );
}

Tailwind CSS provides low-level utility classes that you compose directly in your JSX. Instead of writing CSS, you apply pre-built classes.

üéØ The Tailwind Philosophy

Instead of: .card { padding: 16px; border-radius: 12px; }
You write: className="p-4 rounded-xl"

Part 9: Best Practices & Which to Choose
‚ñº

Making the Right Choice

There's no "one best" styling method. The right choice depends on your project, team, and requirements.

üéØ Decision Guide

If you need... Use...
Quick dynamic styles Inline Styles
Simple project, familiar CSS CSS Files
Scoped styles without extra libraries CSS Modules ‚≠ê Recommended
Complex theming, truly dynamic styles Styled Components
Rapid prototyping, utility-first Tailwind CSS

üìã Best Practices Summary

‚úÖ DO

  • Use CSS variables for theming
  • Co-locate styles with components
  • Use clsx for conditional classes
  • Keep styles scoped when possible
  • Extract reusable styled components
  • Use consistent naming conventions

‚ùå DON'T

  • Mix too many styling methods
  • Use !important frequently
  • Create deeply nested selectors
  • Forget about responsive design
  • Ignore accessibility (color contrast)
  • Over-engineer simple styles

üèÜ Recommended Stack for Most Projects

CSS Modules for component styles
+ Inline styles for dynamic values
+ Global CSS for variables and resets
+ clsx for conditional class names
